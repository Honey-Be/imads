# IMADS: Improved / Integrated Mesh Adaptive Direct Search

> **약어 주석.** 이 저장소에서 **IMADS** 는 **Improved Mesh Adaptive Direct Search** 와 **Integrated Mesh Adaptive Direct Search** 를 함께 가리키는 우산형 약어로 사용된다.

## 초록

본 저장소는 Mesh Adaptive Direct Search(MADS) 계열의 사상을 바탕으로 하되, 현대적 소프트웨어 제약—즉 **다중 fidelity 기반 조기 스크리닝**, **캐시 정합성**, **resumable evaluation**, **정책 레이어의 모듈성**—을 중심에 두고 재구성한 결정론적(Deterministic) derivative-free optimization 엔진을 구현한다. 따라서 현재 리비전은 고전적 의미의 단일체(monolithic) MADS 솔버의 단순 재현이라기보다, \((\tau, S)\) 2축 fidelity ladder, 단계별 후보 승격, 결정론적 audit 선택, filter-plus-progressive-barrier acceptance, 그리고 보수적 스크리닝 임계치의 온라인 보정을 통합한 하나의 **IMADS 구현체**로 읽는 편이 정확하다.

이 구현은 두 가지 목표를 중심으로 조직되어 있다. 첫째, 정책 경계(policy boundary)를 통과하는 모든 연산은 재현 가능해야 한다. 이를 위해 evaluation cache key는 \((x, \phi, \mathrm{env\_rev})\) 로, decision cache key는 여기에 \(\mathrm{policy\_rev}\) 를 추가한 형태로 정의되며, 모든 정책 업데이트는 결정론적 batch boundary 에서만 수행된다. 둘째, 저비용 partial 정보는 탐색 가속을 위해 적극 활용하되, feasibility 확정이나 acceptance 확정에는 사용하지 않는다. 특히 현재 패치본은 두 방향으로 코드베이스를 실질적으로 강화한다. (i) **paired-audit K 2차 고도화**: 같은 \(x\), 같은 \(S\), 여러 tighter-\(\tau\) checkpoint 를 구조적으로 누적하여 K 를 학습한다. (ii) **a 업데이트 정교화**: recency weighting, Wilson confidence interval, false infeasible 통계, precision signal 을 함께 사용하여 early infeasible cut 의 보수성을 조절한다.

또한 현재 코드베이스는 Cargo workspace 구조로 조직되어 있으며, 핵심 엔진(`imads-core`)을 중심으로 C/C++ FFI(`imads-ffi`), JNI(`imads-jni`), Python/PyO3(`imads-py`), WASM(`imads-wasm`) 바인딩 크레이트를 포함한다. 이를 통해 Rust 이외에도 Python(CPython/GraalPython), Java, Kotlin, Scala 3, Clojure, TypeScript, C/C++ 등 다양한 언어 환경에서 동일한 엔진을 사용할 수 있다.

따라서 IMADS 는 단순한 직접 탐색 구현이라기보다, 재현성·적응적 스크리닝·보수적 derivative-free search 의 상호작용을 명시적으로 드러내는 결정론적·정책분리형(policy-separable)·다중 fidelity MADS 엔진으로 이해하는 것이 적절하다.

## 1. 알고리즘 개요

### 1.1 문제 설정

엔진이 다루는 대상은 다음과 같은 black-box 제약 최적화 문제이다.

\[
\min_x f(x) \quad \text{subject to} \quad c_j(x) \le 0, \qquad j=1,\dots,m.
\]

미분은 존재하지 않거나, 존재하더라도 사용하지 않는다는 가정이 기본이다. 구현은 연속공간 기반의 순진한 solver 라기보다 **mesh state** 중심 구조를 취한다. 후보점은 기본 격자 위의 정준 좌표 `XMesh` 로 표현되며, 연속 좌표 `XReal` 은 체크된 사상(mapping)을 통해서만 얻어진다. 이는 수렴성과 결정성에 핵심적인 연산을 부동소수점 위치 자체가 아니라 mesh geometry 위에서 정의하려는 MADS의 기본 관점과 일치한다.

### 1.2 결정론적 실행 모델

본 코드베이스의 핵심은 결정성을 "가능하면 유지하는 성질"이 아니라 **의미론적 요구사항**으로 취급한다는 점이다. 이를 위해 세 가지 장치가 사용된다.

1. **환경 revisioning**: `EnvRev` 는 run identifier, configuration hash, data snapshot identifier, master seed 로부터 결정론적으로 계산된다.
2. **정책 revisioning**: 정책 판정을 바꿀 수 있는 calibration state 변화는 반드시 `PolicyRev` 증가로 이어지며, 이를 통해 obsolete decision cache 의 재사용을 차단한다.
3. **배치 경계 업데이트**: audit, calibrator, DIDS assignment 업데이트는 wall-clock completion order 에 의존하지 않고, batch boundary 에서 정렬된 이벤트 열에 대해서만 적용된다.

이 구조 덕분에 저장소는 concurrency, caching, policy customization 을 동시에 지원하면서도 replayability 를 유지할 수 있다.

### 1.3 AdaptiveExecutor 와 단일 API

`DefaultBundle` 은 `AdaptiveExecutor` 를 사용하며, `workers` 파라미터 하나로 실행 모드를 결정한다.

- `workers = 1`: `InlineExecutor` (순차, 오버헤드 없음)
- `workers > 1`: `WorkerPoolExecutor` (고정 크기 스레드 풀, 배치 배리어 동기화)

`AdaptiveExecutor` 는 `#[cfg(imads_has_threads)]` 게이트로 스레드 가용성을 빌드 타임에 결정한다. native 타겟과 `wasm32-wasip1-threads`, `wasm32-wasip3` 에서는 풀 variant 가 활성화되고, `wasm32-unknown-unknown` 등 스레드 미지원 타겟에서는 인라인 variant 만 사용된다. 이를 통해 사용자는 제네릭 번들 타입을 변경하지 않고도, 단일 API 로 싱글 스레드와 멀티 스레드 실행을 전환할 수 있다.

### 1.4 2축 fidelity ladder

고전적 MADS 위에 본 구현은

\[
\phi = (\tau, S)
\]

로 인덱싱되는 fidelity ladder 를 얹는다. 여기서 \(\tau\) 는 solver tolerance scale, \(S\) 는 Monte Carlo sample size 를 의미한다. 기본 ladder 정책(`StaircaseLadder`)은 대체로

\[
(\tau_1,S_1) \rightarrow (\tau_1,S_2) \rightarrow (\tau_2,S_2) \rightarrow \cdots
\]

형태의 staircase 를 만든다. 즉, \(\tau\) 는 loose 에서 tight 로, \(S\) 는 small 에서 large 로 이동한다. 이 설계는 두 가지 장점을 갖는다.

- 저비용 정보를 먼저 사용하고, 비싼 truth evaluation 은 늦게 구매하게 한다.
- **같은 \(S\), 더 tight 한 \(\tau\)** 비교를 명시적으로 드러내므로, 이번 패치에서 핵심이 된 paired-audit K 학습에 직접적으로 유리하다.

엔진은 ladder 의 마지막 level 을 **TRUTH** 로 간주한다. 그 이전 level 들은 모두 `Estimates` 만을 생성하며, acceptance 를 직접 갱신하지 못한다.

### 1.5 Search, quantization, poll

기본 search 정책은 의도적으로 보수적이다. incumbent 주변에서의 결정론적 local exploration 과, iteration 이 진행될수록 반경이 천천히 증가하는 결정론적 global exploration 을 혼합한다. 생성된 raw proposal 은 모두 현재 mesh 위로 quantize 되며, 이 과정을 통해 canonical coordinate equality 와 cache coherence 가 확보된다.

mesh 상태는 `MeshGeometry` 로 표현되며, 핵심 파라미터는 다음과 같다.

- `base_step`: 기본 격자 간격 \(\Delta_0\),
- `mesh_mul`: 현재 mesh multiplier,
- `refine_div`: 정수 기반 refinement divisor,
- `poll_step_mult`: base-lattice 단위 poll-step multiplier.

따라서 현재 mesh spacing 은

\[
\Delta = \Delta_0 \cdot \texttt{mesh\_mul}
\]

가 되며, refinement 는 정수 나눗셈으로 수행된다. 이로써 nested mesh 성질이 구조적으로 유지된다. 기본 poll 연산자(`DefaultPoll`)는 incumbent mesh point 주변의 축방향 signed coordinate direction 들을 생성한다. acceptance 규칙은 봉인(sealed)되어 있지만, poll 자체는 투명하고 단순한 형태를 유지한다.

## 2. Partial 정보, 보수적 스크리닝, 그리고 Acceptance

### 2.1 Partial estimates

truth 이전 fidelity level 에서 evaluator 는 `Estimates` 객체를 반환한다. 여기에는 다음이 포함된다.

- 목적함수 추정치 \(\hat f\) 와 표준오차,
- 제약 추정치 \(\hat c_j\) 와 표준오차,
- 해당 추정치를 생성한 fidelity bucket \(\phi=(\tau,S)\),
- solver-induced bias 모델에 사용되는 `tau_scale`.

기본 toy evaluator 는 의도된 계약을 잘 보여준다. Monte Carlo sample 은 \((x,\phi,\mathrm{env},k)\) 에 대해 결정론적이며, solver bias 는 \(\tau\)-의존적 결정론적 교란으로 표현된다. 엔진은 그 이상의 내부 구조를 가정하지 않는다.

### 2.2 Margin policy

margin policy 는 calibrator state 를 실제 스크리닝 임계치로 변환한다. 현재 구현에서 constraint \(j\) 에 대한 early infeasible 선언은 다음 부등식을 만족할 때에만 이루어진다.

\[
\hat c_j - z_{\alpha_c}\,\hat\sigma_{c_j} - K_{c,j}(\phi)\,\tau_{\mathrm{scale}} > \delta^{\mathrm{rel}}_j.
\]

여기서 세 가지 점이 중요하다.

1. 검정은 **단측(one-sided)** 이며 **보수적**이다.
2. effective bias coefficient 는 더 이상 완전히 global 하지 않다. 이번 패치 이후 구현은 가능한 경우 먼저 **\((\tau,S)\)-bucket-specific** 값 `k_c_for(phi, j)` 를 사용하고, 없을 때만 global coefficient 로 fallback 한다.
3. 임계치 벡터 \(\delta^{\mathrm{rel}}\) 는 constraint 별로 유지되며 적응적으로 보정된다.

목적함수 프루닝 또한 calibration-aware conservative bound 테스트로 구성된다.

\[
\mathrm{LCB}_f = \hat f - z_{\alpha_f}\hat\sigma_f - K_f(\phi)\,\tau_{\mathrm{scale}},
\qquad
\mathrm{UCB}_f = \hat f + z_{\beta_f}\hat\sigma_f + K_f(\phi)\,\tau_{\mathrm{scale}}.
\]

현재 구현은 가능하면 `k_f_for(phi)` 를 사용한다. 따라서 objective pruning 은 calibration 과 무관한 임시 휴리스틱이 아니라, calibration state 를 직접 반영한 보수적 하한 검사이다.

### 2.3 Truth-only acceptance

acceptance logic 은 의도적으로 봉인되어 있으며, **filter plus progressive barrier** 구조를 취한다. non-truth result 는 scheduling 과 promotion 에는 영향을 줄 수 있지만, filter 에 point 를 삽입하거나 feasibility 를 확정할 수는 없다. 오직 `Truth` 결과만 acceptance 를 갱신할 수 있다.

이 설계는 다음 두 행위를 엄격히 분리한다.

- **screening**: 명백히 지배되거나 명백히 infeasible 한 후보를 조기에 중단하는 행위,
- **acceptance**: terminal fidelity 에서만 feasible incumbent 와 filter 를 갱신하는 행위.

소프트웨어 정합성 관점에서 이 분리는 매우 중요하다. partial estimate 가 acceptance 구조를 직접 바꾸기 시작하면 cache semantics, policy revisioning, replayability 를 동시에 유지하기가 급격히 어려워진다.

## 3. Audited Early Infeasible Cut 과 Paired-Audit K 2차 고도화

### 3.1 Audit 이 필요한 이유

early infeasible cut 은 비용 측면에서는 매력적이지만 통계적으로는 위험하다. cut 이 지나치게 공격적이면 truth 에서 feasible 로 판명될 점들이 조기에 제거될 수 있다. IMADS 는 이를 방지하기 위해 결정론적 audit policy 를 둔다. audit 대상으로 선택된 early cut 은 즉시 종료되지 않고, 더 높은 fidelity 로 강제 승격된다.

selector 자체는 hash 기반이므로 재현 가능하다. 그러나 더 본질적인 것은 selector 가 아니라 **audit path 의 구조**이다.

### 3.2 단일 paired sample 에서 다중 \(\tau\) checkpoint 로

이번 패치 이전의 K 학습은 사실상 1차 버전에 가까웠다. 하나의 early-cut estimate 는 많아야 하나의 paired checkpoint 와만 연결되었다. 패치 이후 구현은 이를 실질적으로 일반화한다.

어떤 audited cut 이 ladder index `phi_idx_at_cut` 에서 발생하면, 엔진은 이후 ladder 상의 level 들 중 다음 조건을 만족하는 모든 index 를 수집한다.

- 같은 candidate \(x\),
- 같은 Monte Carlo budget \(S\),
- 더 tight 한 solver tolerance \(\tau' < \tau\).

이 index 들은 `CandidateAuditOrigin::paired_phi_indices` 에 저장되며, 구조화된 audit subpath 를 정의한다. 따라서 candidate 는 단순히 truth 로 점프하는 것이 아니라, 여러 same-\(S\), tighter-\(\tau\) checkpoint 를 통과하면서 정보를 누적할 수 있다.

### 3.3 구조화된 K 누적

truth 에 도달하면 calibrator 는

\[
(\tau^{(0)},S,\hat y^{(0)}) \, ,\, (\tau^{(1)},S,\hat y^{(1)}) \, ,\, \dots \, ,\, (\tau^{(r)},S,\hat y^{(r)})
\]

형태의 checkpoint 열을 재구성한다. 여기서 \(\hat y\) 는 목적함수·제약 추정치를 함께 가리킨다. 이후 calibrator 는 단일 loose-vs-truth 비교만을 사용하지 않고, 인접 checkpoint 간의 interval pair 를 형성한다. 각 인접 pair 에 대해 다음과 같은 tau-normalized positive delta 가 계산된다.

\[
\frac{(\hat f^{\text{loose}}-\hat f^{\text{tight}})_+}{\tau^{\text{loose}}-\tau^{\text{tight}}},
\qquad
\frac{(\hat c_j^{\text{loose}}-\hat c_j^{\text{tight}})_+}{\tau^{\text{loose}}-\tau^{\text{tight}}}.
\]

이 값들은 세 수준의 통계 뷰로 축적된다.

1. **global K samples**: 전체 엔진 차원의 `k_f`, `k_c[j]` 업데이트에 사용된다.
2. **phi-bucketed K samples**: loose endpoint \((\tau,S)\) 기준으로 모아 `k_by_phi` 를 업데이트한다.
3. **interval-bucketed K samples**: \((\tau_{\text{loose}},\tau_{\text{tight}},S)\) 기준으로 모아 `k_interval_stats` 로 노출된다.

구조화된 same-\(S\) checkpoint 가 전혀 없을 때에는 cut-to-truth fallback 이 사용된다. 즉, 기능은 유지하되, preferred path 는 same-\(S\) 기반 interval accumulation 으로 분명히 구분된다.

### 3.4 Quantile-smoothed K 업데이트

K 업데이트는 두 겹으로 보수적이다. 첫째, loose-minus-tight delta 의 양수 부분만 누적하므로, 학습되는 K 는 unbiased difference estimator 라기보다 upper-envelope surrogate 에 가깝다. 둘째, K 는 평균이 아니라 높은 empirical quantile 의 지수이동평균으로 갱신된다.

\[
K^{\text{new}} = (1-\eta_K)K^{\text{old}} + \eta_K Q_q.
\]

여기서 \(Q_q\) 는 최근 sample 들의 empirical \(q\)-quantile 이다. margin policy 가 K 를 단측 스크리닝 부등식 안에서 사용한다는 점을 생각하면, 이런 선택은 "전형적" bias 를 추적하기보다 **안전 측면에서 충분히 큰 bias scale** 을 추적하려는 설계로 해석하는 것이 타당하다.

## 4. 정교화된 DIDS Assignment Update

### 4.1 \(a_j\) 의 의미

DIDS policy 는 \(a_j \in \{1,\dots,L\}\) 형태의 assignment vector 를 유지한다. \(L\) 은 ladder length 이다. 그 의미는 단순하다.

> constraint \(j\) 에 대한 early infeasible termination 은 1-based ladder index 가 \(a_j\) 이상인 fidelity level 에서만 허용된다.

따라서 \(a_j\) 가 커질수록 더 보수적이다. assignment policy 는 결국 "cut 을 신뢰하기 전에 얼마나 높은 fidelity 를 구매해야 하는가?"라는 비용-안전성 문제를 중재한다.

### 4.2 Recency-weighted base rule

업데이트 1단계는 기록된 early infeasible outcome 들로부터 constraint 별 적절한 cutting level 을 추정한다. 패치 이후에는 단순 빈도 집계가 아니라, exponential recency factor 를 부여한 후 weighted upper quantile 로 base level 을 잡는다. 이로써 오래된 데이터에 하드하게 잘리는 슬라이딩 윈도우 없이도, 최근 관측이 자연스럽게 더 큰 영향력을 갖는다.

### 4.3 Confidence-aware false infeasible 제어

업데이트 2단계는 calibrator 가 유지하는 audit 통계를 사용한다. 각 ladder level 과 constraint 에 대해 다음 두 양에 대한 Wilson confidence bound 를 계산한다.

- false infeasible rate,
- 원래 flag 된 constraint 가 truth 에서도 실제로 위반되었는지에 대한 precision signal.

false infeasible rate 의 Wilson upper bound 가 너무 크거나, precision 의 Wilson lower bound 가 너무 낮으면 \(a_j\) 는 상승한다. 즉, 코드가 더 이상 점추정치(point estimate)만 보고 반응하지 않고, **finite-sample uncertainty 를 명시적으로 반영한 보수적 통계량**에 의해 반응한다는 점이 이번 패치의 핵심이다.

### 4.4 Relaxation logic

반대로 \(a_j\) 를 낮추는 규칙도 훨씬 엄격해졌다. 바로 아래 level 에 충분한 audit evidence 가 축적되어 있어야 하고, false infeasible upper bound 가 목표치보다 충분히 낮아야 하며, precision lower bound 도 local 및 overall 수준에서 모두 적절해야 한다. 요컨대, 새 정책은 "최근 숫자가 그럴듯해 보인다"는 이유만으로 보수성을 낮추지 않고, **통계적으로 정당화될 때에만** relax 를 허용한다.

### 4.5 \(\delta^{\mathrm{rel}}\) 과의 상호작용

마지막 fallback guard 는 `delta_rel[j]` 가 baseline 대비 유난히 커진 경우를 "이 constraint 는 안전하게 screen 하기 어려워졌다"는 신호로 해석한다. `delta_rel[j] / delta_baseline` 비율이 충분히 커지면 \(a_j\) 에 추가 상승 압력이 가해진다. 즉, DIDS logic 이 calibrator 의 false-infeasible control loop 와 분리되지 않고 다시 연결된다.

## 5. 엔진 의미론과 소스 구조

### 5.1 Candidate lifecycle

각 candidate 는 `CandidateStageState` 로 보관되며, 여기에는 다음 정보가 들어 있다.

- 현재 ladder index,
- 마지막으로 확보한 estimate,
- audit flag 및 audit origin,
- 제출 당시의 policy revision snapshot,
- incumbent snapshot identifier,
- terminal 또는 resumable status.

이 덕분에 엔진은 새로 제안된 후보와 이전 iteration 에서 중단되었던 후보를 섞어 처리하더라도, 나중의 판단이 어떤 전제 위에서 유효한지를 잃지 않는다.

### 5.2 Cache 분리

저장소는 캐시를 두 층으로 나눈다.

- `EvalCache`: \((x,\phi,\mathrm{env\_rev})\) 를 key 로 하는 expensive black-box estimate 캐시,
- `DecisionCache`: \((x,\phi,\mathrm{env\_rev},\mathrm{policy\_rev},\mathrm{tag})\) 를 key 로 하는 policy-dependent materialized outcome 캐시.

이 분리는 단순한 구현 편의가 아니다. 정책이 바뀌어도 evaluation reuse 는 유지하면서, policy-sensitive decision 은 obsolete calibration 체제 아래서 재생되지 않도록 하기 위해 꼭 필요하다.

### 5.3 Workspace 구조와 주요 소스 파일

코드베이스는 Cargo workspace 로 조직되어 있다.

- **`imads-core/`**: 핵심 엔진 크레이트
- **`imads-ffi/`**: C/C++ FFI (opaque handle + `cbindgen` 헤더)
- **`imads-jni/`**: JVM 언어용 JNI 바인딩 (Java bridge 포함)
- **`imads-py/`**: Python 바인딩 (PyO3/maturin, CPython + GraalPython)
- **`imads-wasm/`**: WASM/TypeScript 바인딩 (wasm-bindgen, bundler/web/nodejs 타겟)

핵심 엔진 크레이트(`imads-core/src/`)는 대체로 다음 순서로 읽는 것이 가장 이해하기 쉽다.

1. `types.rs`: 정준 상태 객체, fidelity type, cache key, candidate lifecycle metadata.
2. `core/engine.rs`: 결정론적 step-wise engine, `AdaptiveExecutor`, materialization logic, audit-path routing, policy-boundary update.
3. `policies/margin.rs`: conservative early infeasible test 와 objective bound 계산.
4. `policies/calibrator.rs`: delta control, paired-audit bookkeeping, K quantile update, bucketed statistics.
5. `policies/dids.rs`: early infeasible termination level 을 정하는 adaptive assignment rule.
6. `core/acceptance.rs`: sealed filter-plus-progressive-barrier acceptance.
7. `policies/ladder.rs`, `policies/search.rs`, `core/poll.rs`: ladder 구성, 후보 제안, poll geometry.

### 5.4 현재 구현의 성격

현재 코드베이스는 **잘 정의된 policy surface 를 가진 결정론적 IMADS 엔진**으로 이해하는 것이 가장 적절하다. calibrated screening, paired-audit K logic, resumable state transition, 다중 언어 FFI 지원 등 실질적인 알고리즘 및 인프라 내용을 이미 상당 부분 갖추고 있으나, 이를 곧바로 대규모 성숙 DFO 패키지의 완전한 대체물로 제시하는 것은 과도하다. 이 구현의 핵심 가치는, MADS 계열 탐색·다중 fidelity 스크리닝·결정성·캐시 정합성 사이의 상호작용을 명시적이고 검토 가능한 형태로 드러낸다는 데 있다.
