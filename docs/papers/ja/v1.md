# IMADS: Improved / Integrated Mesh Adaptive Direct Search

> **略語に関する注記.** 本リポジトリにおいて **IMADS** は **Improved Mesh Adaptive Direct Search** と **Integrated Mesh Adaptive Direct Search** の両方を包括する傘型略語として使用される。

## 概要

本リポジトリは、Mesh Adaptive Direct Search（MADS）系列の思想を基盤としつつ、現代的なソフトウェア制約――すなわち**多重 fidelity に基づく早期スクリーニング**、**キャッシュ整合性**、**resumable evaluation**、**ポリシーレイヤのモジュール性**――を中心に再構成した決定論的（Deterministic）derivative-free optimization エンジンを実装するものである。したがって現在のリビジョンは、古典的な意味での一体型（monolithic）MADS ソルバの単純な再現というよりも、\((\tau, S)\) 2軸 fidelity ladder、段階的候補昇格、決定論的 audit 選択、filter-plus-progressive-barrier acceptance、そして保守的スクリーニング閾値のオンライン較正を統合した一つの **IMADS 実装体**として読む方が正確である。

本実装は二つの目標を中心に組織されている。第一に、ポリシー境界（policy boundary）を通過するすべての演算は再現可能でなければならない。これを実現するため、evaluation cache key は \((x, \phi, \mathrm{env\_rev})\) として、decision cache key はこれに \(\mathrm{policy\_rev}\) を追加した形式で定義され、すべてのポリシー更新は決定論的 batch boundary においてのみ実行される。第二に、低コストの partial 情報は探索加速のために積極的に活用するが、feasibility の確定や acceptance の確定には使用しない。特に現在のパッチ版は二つの方向でコードベースを実質的に強化している。(i) **paired-audit K の2次高度化**：同一の \(x\)、同一の \(S\)、複数のより厳密な \(\tau\) checkpoint を構造的に蓄積して K を学習する。(ii) **a 更新の精緻化**：recency weighting、Wilson confidence interval、false infeasible 統計、precision signal を併用し、early infeasible cut の保守性を調整する。

また、現在のコードベースは Cargo workspace 構造で組織されており、中核エンジン（`imads-core`）を中心に C/C++ FFI（`imads-ffi`）、JNI（`imads-jni`）、Python/PyO3（`imads-py`）、WASM（`imads-wasm`）のバインディングクレートを含む。これにより、Rust 以外にも Python（CPython/GraalPython）、Java、Kotlin、Scala 3、Clojure、TypeScript、C/C++ など多様な言語環境から同一のエンジンを利用することが可能である。

したがって IMADS は、単純な直接探索の実装というよりも、再現性・適応的スクリーニング・保守的 derivative-free search の相互作用を明示的に表出する決定論的・ポリシー分離型（policy-separable）・多重 fidelity MADS エンジンとして理解するのが適切である。

## 1. アルゴリズムの概要

### 1.1 問題設定

エンジンが扱う対象は、以下のような black-box 制約付き最適化問題である。

\[
\min_x f(x) \quad \text{subject to} \quad c_j(x) \le 0, \qquad j=1,\dots,m.
\]

微分は存在しないか、存在したとしても使用しないという仮定が基本である。実装は連続空間に基づく素朴な solver ではなく、**mesh state** 中心の構造を採用している。候補点は基本格子上の正準座標 `XMesh` で表現され、連続座標 `XReal` は検証済みの写像（mapping）を通じてのみ得られる。これは、収束性と決定性にとって核心的な演算を浮動小数点の位置そのものではなく mesh geometry 上で定義しようとする MADS の基本的な観点と一致するものである。

### 1.2 決定論的実行モデル

本コードベースの核心は、決定性を「可能な限り維持する性質」としてではなく、**意味論的要件**として扱う点にある。これを実現するために三つの装置が用いられる。

1. **環境バージョニング**：`EnvRev` は run identifier、configuration hash、data snapshot identifier、master seed から決定論的に計算される。
2. **ポリシーバージョニング**：ポリシー判定を変更し得る calibration state の変化は、必ず `PolicyRev` の増加を伴い、これにより obsolete な decision cache の再利用が阻止される。
3. **バッチ境界更新**：audit、calibrator、DIDS assignment の更新は wall-clock completion order に依存せず、batch boundary において整列されたイベント列に対してのみ適用される。

この構造により、リポジトリは concurrency、caching、policy customization を同時に支援しつつも replayability を維持することが可能である。

### 1.3 AdaptiveExecutor と単一 API

`DefaultBundle` は `AdaptiveExecutor` を使用し、`workers` パラメータ一つで実行モードを決定する。

- `workers = 1`：`InlineExecutor`（逐次実行、オーバーヘッドなし）
- `workers > 1`：`WorkerPoolExecutor`（固定サイズスレッドプール、バッチバリア同期）

`AdaptiveExecutor` は `#[cfg(imads_has_threads)]` ゲートによりスレッドの利用可能性をビルド時に判定する。ネイティブターゲットおよび `wasm32-wasip1-threads`、`wasm32-wasip3` ではプール variant が有効化され、`wasm32-unknown-unknown` 等のスレッド非対応ターゲットではインライン variant のみが使用される。これにより、利用者はジェネリックなバンドル型を変更することなく、単一 API でシングルスレッドとマルチスレッド実行を切り替えることができる。

### 1.4 2軸 fidelity ladder

古典的 MADS の上に、本実装は

\[
\phi = (\tau, S)
\]

でインデクスされる fidelity ladder を重ねる。ここで \(\tau\) は solver tolerance scale、\(S\) は Monte Carlo sample size を意味する。デフォルトの ladder ポリシー（`StaircaseLadder`）は概ね

\[
(\tau_1,S_1) \rightarrow (\tau_1,S_2) \rightarrow (\tau_2,S_2) \rightarrow \cdots
\]

の形の staircase を構成する。すなわち、\(\tau\) は loose から tight へ、\(S\) は small から large へ移動する。この設計には二つの利点がある。

- 低コスト情報を先に使用し、高コストの truth evaluation は後から購入させる。
- **同一の \(S\)、より tight な \(\tau\)** の比較を明示的に表出するため、本パッチで核心となった paired-audit K 学習に直接的に有利である。

エンジンは ladder の最終 level を **TRUTH** とみなす。それ以前の level はすべて `Estimates` のみを生成し、acceptance を直接更新することはできない。

### 1.5 Search、量子化、poll

デフォルトの search ポリシーは意図的に保守的である。incumbent 周辺での決定論的 local exploration と、反復が進むにつれて半径が緩やかに拡大する決定論的 global exploration を混合する。生成された raw proposal はすべて現在の mesh 上に quantize され、この過程を通じて canonical coordinate equality とキャッシュコヒーレンスが確保される。

mesh 状態は `MeshGeometry` で表現され、核心パラメータは以下の通りである。

- `base_step`：基本格子間隔 \(\Delta_0\)、
- `mesh_mul`：現在の mesh multiplier、
- `refine_div`：整数ベースの refinement divisor、
- `poll_step_mult`：base-lattice 単位の poll-step multiplier。

したがって、現在の mesh spacing は

\[
\Delta = \Delta_0 \cdot \texttt{mesh\_mul}
\]

となり、refinement は整数除算で実行される。これにより nested mesh 性質が構造的に維持される。デフォルトの poll 演算子（`DefaultPoll`）は incumbent mesh point 周辺の軸方向の符号付き座標方向を生成する。acceptance 規則は封印（sealed）されているが、poll 自体は透明かつ単純な形態を維持している。

## 2. Partial 情報、保守的スクリーニング、そして Acceptance

### 2.1 Partial estimates

truth 以前の fidelity level において evaluator は `Estimates` オブジェクトを返す。ここには以下が含まれる。

- 目的関数推定値 \(\hat f\) と標準誤差、
- 制約推定値 \(\hat c_j\) と標準誤差、
- 当該推定値を生成した fidelity bucket \(\phi=(\tau,S)\)、
- solver-induced bias モデルに使用される `tau_scale`。

デフォルトの toy evaluator は、意図された契約を明確に示している。Monte Carlo sample は \((x,\phi,\mathrm{env},k)\) に対して決定論的であり、solver bias は \(\tau\) 依存の決定論的擾乱として表現される。エンジンはそれ以上の内部構造を仮定しない。

### 2.2 Margin policy

margin policy は calibrator state を実際のスクリーニング閾値に変換する。現在の実装において、制約 \(j\) に対する early infeasible 宣言は、以下の不等式が満たされた場合にのみ行われる。

\[
\hat c_j - z_{\alpha_c}\,\hat\sigma_{c_j} - K_{c,j}(\phi)\,\tau_{\mathrm{scale}} > \delta^{\mathrm{rel}}_j.
\]

ここで三つの点が重要である。

1. 検定は**片側（one-sided）**であり**保守的**である。
2. effective bias coefficient はもはや完全に global ではない。本パッチ以降の実装は、可能な場合にまず **\((\tau,S)\)-bucket-specific** な値 `k_c_for(phi, j)` を使用し、存在しない場合にのみ global coefficient へ fallback する。
3. 閾値ベクトル \(\delta^{\mathrm{rel}}\) は制約ごとに維持され、適応的に較正される。

目的関数プルーニングもまた、calibration-aware な保守的 bound テストとして構成される。

\[
\mathrm{LCB}_f = \hat f - z_{\alpha_f}\hat\sigma_f - K_f(\phi)\,\tau_{\mathrm{scale}},
\qquad
\mathrm{UCB}_f = \hat f + z_{\beta_f}\hat\sigma_f + K_f(\phi)\,\tau_{\mathrm{scale}}.
\]

現在の実装は、可能な場合には `k_f_for(phi)` を使用する。したがって objective pruning は calibration と無関係な一時的ヒューリスティックではなく、calibration state を直接反映した保守的下界検査である。

### 2.3 Truth-only acceptance

acceptance logic は意図的に封印されており、**filter plus progressive barrier** 構造を採用している。non-truth の結果は scheduling と promotion に影響を与え得るが、filter に point を挿入したり feasibility を確定したりすることはできない。`Truth` の結果のみが acceptance を更新できる。

この設計は以下の二つの行為を厳格に分離する。

- **screening**：明らかに劣位にあるか明らかに infeasible な候補を早期に打ち切る行為、
- **acceptance**：terminal fidelity においてのみ feasible incumbent と filter を更新する行為。

ソフトウェア整合性の観点から、この分離は極めて重要である。partial estimate が acceptance 構造を直接変更し始めると、cache semantics、policy revisioning、replayability を同時に維持することが急激に困難になる。

## 3. Audited Early Infeasible Cut と Paired-Audit K の2次高度化

### 3.1 Audit が必要な理由

early infeasible cut はコスト面では魅力的であるが、統計的には危険である。cut が過度に攻撃的であると、truth において feasible と判明するはずの点が早期に除去される可能性がある。IMADS はこれを防止するために決定論的 audit policy を設ける。audit 対象として選択された early cut は即座に終了されず、より高い fidelity へと強制昇格される。

selector 自体は hash ベースであるため再現可能である。しかし、より本質的なのは selector ではなく **audit path の構造**である。

### 3.2 単一 paired sample から多重 \(\tau\) checkpoint へ

本パッチ以前の K 学習は、実質的に第1次バージョンに近いものであった。一つの early-cut estimate は、多くとも一つの paired checkpoint とのみ接続されていた。パッチ以降の実装はこれを実質的に一般化する。

ある audited cut が ladder index `phi_idx_at_cut` で発生した場合、エンジンはその後の ladder 上の level のうち以下の条件を満たすすべての index を収集する。

- 同一の candidate \(x\)、
- 同一の Monte Carlo budget \(S\)、
- より厳密な solver tolerance \(\tau' < \tau\)。

これらの index は `CandidateAuditOrigin::paired_phi_indices` に格納され、構造化された audit subpath を定義する。したがって candidate は単純に truth へジャンプするのではなく、複数の same-\(S\)、tighter-\(\tau\) checkpoint を通過しながら情報を蓄積することができる。

### 3.3 構造化された K の蓄積

truth に到達すると、calibrator は

\[
(\tau^{(0)},S,\hat y^{(0)}) \, ,\, (\tau^{(1)},S,\hat y^{(1)}) \, ,\, \dots \, ,\, (\tau^{(r)},S,\hat y^{(r)})
\]

の形の checkpoint 列を再構成する。ここで \(\hat y\) は目的関数・制約推定値を併せて指す。その後 calibrator は、単一の loose-vs-truth 比較のみを使用するのではなく、隣接 checkpoint 間の interval pair を形成する。各隣接 pair に対して、以下のような tau-normalized positive delta が計算される。

\[
\frac{(\hat f^{\text{loose}}-\hat f^{\text{tight}})_+}{\tau^{\text{loose}}-\tau^{\text{tight}}},
\qquad
\frac{(\hat c_j^{\text{loose}}-\hat c_j^{\text{tight}})_+}{\tau^{\text{loose}}-\tau^{\text{tight}}}.
\]

これらの値は三つの水準の統計ビューに蓄積される。

1. **global K samples**：エンジン全体レベルの `k_f`、`k_c[j]` の更新に使用される。
2. **phi-bucketed K samples**：loose endpoint \((\tau,S)\) を基準にまとめて `k_by_phi` を更新する。
3. **interval-bucketed K samples**：\((\tau_{\text{loose}},\tau_{\text{tight}},S)\) を基準にまとめて `k_interval_stats` として公開される。

構造化された same-\(S\) checkpoint が全く存在しない場合には cut-to-truth fallback が使用される。すなわち、機能は維持されるが、preferred path は same-\(S\) ベースの interval accumulation として明確に区別される。

### 3.4 Quantile-smoothed K 更新

K の更新は二重に保守的である。第一に、loose-minus-tight delta の正の部分のみを蓄積するため、学習される K は unbiased difference estimator というよりも upper-envelope surrogate に近い。第二に、K は平均ではなく高い empirical quantile の指数移動平均として更新される。

\[
K^{\text{new}} = (1-\eta_K)K^{\text{old}} + \eta_K Q_q.
\]

ここで \(Q_q\) は最近の sample 群の empirical \(q\)-quantile である。margin policy が K を片側スクリーニング不等式の中で使用するという点を考慮すると、この選択は「典型的な」bias を追跡するというよりも、**安全側において十分に大きな bias scale** を追跡しようとする設計として解釈するのが妥当である。

## 4. 精緻化された DIDS Assignment Update

### 4.1 \(a_j\) の意味

DIDS policy は \(a_j \in \{1,\dots,L\}\) の形の assignment vector を維持する。ここで \(L\) は ladder length である。その意味は単純である。

> 制約 \(j\) に対する early infeasible termination は、1-based ladder index が \(a_j\) 以上の fidelity level においてのみ許容される。

したがって \(a_j\) が大きくなるほどより保守的となる。assignment policy は結局「cut を信頼する前にどれだけ高い fidelity を購入すべきか」というコスト・安全性の問題を仲裁するものである。

### 4.2 Recency-weighted base rule

更新の第1段階は、記録された early infeasible outcome から制約ごとの適切な cutting level を推定する。パッチ以降は単純な頻度集計ではなく、exponential recency factor を付与した上で weighted upper quantile により base level を決定する。これにより、古いデータに対してハードにカットされるスライディングウィンドウを用いずとも、最近の観測が自然とより大きな影響力を持つようになる。

### 4.3 Confidence-aware false infeasible 制御

更新の第2段階は、calibrator が維持する audit 統計を使用する。各 ladder level と制約に対して、以下の二つの量の Wilson confidence bound を計算する。

- false infeasible rate、
- 元々フラグされた制約が truth においても実際に違反されていたかに関する precision signal。

false infeasible rate の Wilson upper bound が大きすぎるか、precision の Wilson lower bound が低すぎる場合、\(a_j\) は上昇する。すなわち、コードがもはや点推定値（point estimate）のみに基づいて反応するのではなく、**有限標本の不確実性を明示的に反映した保守的統計量**に基づいて反応するという点が、本パッチの核心である。

### 4.4 Relaxation logic

逆に \(a_j\) を低下させる規則も、はるかに厳格になった。直下の level に十分な audit evidence が蓄積されていなければならず、false infeasible upper bound が目標値よりも十分に低くなければならず、precision lower bound も local および overall の両水準で適切でなければならない。要するに、新しいポリシーは「最近の数値がもっともらしく見える」という理由だけで保守性を低下させることはなく、**統計的に正当化された場合にのみ** relax を許容する。

### 4.5 \(\delta^{\mathrm{rel}}\) との相互作用

最後の fallback guard は、`delta_rel[j]` が baseline に対して異常に大きくなった場合を「この制約は安全にスクリーニングすることが困難になった」というシグナルとして解釈する。`delta_rel[j] / delta_baseline` の比率が十分に大きくなると、\(a_j\) に追加の上昇圧力が加わる。すなわち、DIDS logic は calibrator の false-infeasible control loop から分離されるのではなく、再び接続される。

## 5. エンジンのセマンティクスとソース構造

### 5.1 Candidate lifecycle

各 candidate は `CandidateStageState` として保持され、ここには以下の情報が含まれる。

- 現在の ladder index、
- 最後に取得した estimate、
- audit flag および audit origin、
- 提出時の policy revision snapshot、
- incumbent snapshot identifier、
- terminal または resumable status。

これにより、エンジンは新たに提案された候補と以前のイテレーションで中断された候補を混在させて処理する場合でも、後の判断がどのような前提の上で有効であるかを失わない。

### 5.2 Cache の分離

リポジトリはキャッシュを二層に分離する。

- `EvalCache`：\((x,\phi,\mathrm{env\_rev})\) を key とする高コスト black-box estimate キャッシュ、
- `DecisionCache`：\((x,\phi,\mathrm{env\_rev},\mathrm{policy\_rev},\mathrm{tag})\) を key とする policy-dependent な materialized outcome キャッシュ。

この分離は単なる実装上の便宜ではない。ポリシーが変更されても evaluation reuse は維持しつつ、policy-sensitive な decision が obsolete な calibration 体制の下で再生されないようにするために不可欠である。

### 5.3 Workspace 構造と主要ソースファイル

コードベースは Cargo workspace として組織されている。

- **`imads-core/`**：中核エンジンクレート
- **`imads-ffi/`**：C/C++ FFI（opaque handle + `cbindgen` ヘッダ）
- **`imads-jni/`**：JVM 言語用 JNI バインディング（Java bridge 含む）
- **`imads-py/`**：Python バインディング（PyO3/maturin、CPython + GraalPython）
- **`imads-wasm/`**：WASM/TypeScript バインディング（wasm-bindgen、bundler/web/nodejs ターゲット）

中核エンジンクレート（`imads-core/src/`）は、概ね以下の順序で読むのが最も理解しやすい。

1. `types.rs`：正準状態オブジェクト、fidelity type、cache key、candidate lifecycle metadata。
2. `core/engine.rs`：決定論的 step-wise engine、`AdaptiveExecutor`、materialization logic、audit-path routing、policy-boundary update。
3. `policies/margin.rs`：保守的 early infeasible テストと objective bound 計算。
4. `policies/calibrator.rs`：delta 制御、paired-audit bookkeeping、K quantile 更新、bucketed statistics。
5. `policies/dids.rs`：early infeasible termination level を決定する適応的 assignment rule。
6. `core/acceptance.rs`：封印された filter-plus-progressive-barrier acceptance。
7. `policies/ladder.rs`、`policies/search.rs`、`core/poll.rs`：ladder 構成、候補提案、poll geometry。

### 5.4 現在の実装の性格

現在のコードベースは、**明確に定義された policy surface を持つ決定論的 IMADS エンジン**として理解するのが最も適切である。calibrated screening、paired-audit K logic、resumable state transition、多言語 FFI 対応など、実質的なアルゴリズムおよびインフラストラクチャの内容を既にかなりの程度備えているが、これを直ちに大規模な成熟 DFO パッケージの完全な代替物として提示することは過大である。本実装の核心的価値は、MADS 系列の探索・多重 fidelity スクリーニング・決定性・キャッシュ整合性の間の相互作用を、明示的かつ検証可能な形で表出する点にある。
