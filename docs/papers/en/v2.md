# IMADS: Improved / Integrated Mesh Adaptive Direct Search

> **Abbreviation note.** In this repository, **IMADS** is used as an umbrella abbreviation that refers to both **Improved Mesh Adaptive Direct Search** and **Integrated Mesh Adaptive Direct Search**.

## Abstract

This repository implements a deterministic derivative-free optimization engine that draws on the Mesh Adaptive Direct Search (MADS) family of ideas while being reorganized around modern software constraints -- namely **multi-fidelity early screening**, **cache coherence**, **resumable evaluation**, and **modularity of the policy layer**. Accordingly, the current revision is more accurately read not as a straightforward reimplementation of a classical monolithic MADS solver, but rather as an **IMADS implementation** that integrates a \((\tau, S)\) two-axis fidelity ladder, stage-wise candidate promotion, deterministic audit selection, filter-plus-progressive-barrier acceptance, and online calibration of conservative screening thresholds.

This implementation is organized around two goals. First, every operation that crosses a policy boundary must be reproducible. To this end, evaluation cache keys are defined as \((x, \phi, \mathrm{env\_rev})\), decision cache keys augment these with \(\mathrm{policy\_rev}\), and all policy updates are performed only at deterministic batch boundaries. Second, low-cost partial information is aggressively exploited to accelerate the search, but is never used to finalize feasibility or acceptance decisions. In particular, the current patch substantively strengthens the codebase in two directions: (i) **paired-audit K second-order refinement**: K is learned by structurally accumulating multiple tighter-\(\tau\) checkpoints at the same \(x\) and the same \(S\); and (ii) **refined a update**: recency weighting, Wilson confidence intervals, false infeasible statistics, and precision signals are jointly used to modulate the conservatism of early infeasible cuts.

Furthermore, the current codebase is organized as a Cargo workspace, centered on the core engine (`imads-core`), with binding crates for C/C++ FFI (`imads-ffi`), JVM via FFM (`imads-jvm`), Python/PyO3 (`imads-py`), and WASM via Component Model (`imads-wasm`). This enables the same engine to be used from a variety of language environments beyond Rust, including Python (CPython/GraalPython), Java, Kotlin, Scala 3, Clojure, TypeScript, and C/C++.

IMADS is therefore best understood not as a simple direct search implementation, but as a deterministic, policy-separable, multi-fidelity MADS engine that explicitly exposes the interactions among reproducibility, adaptive screening, and conservative derivative-free search.

## 1. Algorithm Overview

### 1.1 Problem Setting

The engine addresses the following black-box constrained optimization problem:

\[
\min_x f(x) \quad \text{subject to} \quad c_j(x) \le 0, \qquad j=1,\dots,m.
\]

The standing assumption is that derivatives either do not exist or are not used even if they do. The implementation adopts a **mesh state**-centric architecture rather than that of a naive continuous-space solver. Candidate points are represented as canonical coordinates `XMesh` on the base lattice, and continuous coordinates `XReal` are obtained only through a checked mapping. This is consistent with the fundamental MADS perspective of defining convergence-critical and determinism-critical operations on mesh geometry rather than on floating-point positions themselves.

### 1.2 Deterministic Execution Model

A central tenet of this codebase is that determinism is treated not as a property to be maintained "when possible," but as a **semantic requirement**. Three mechanisms are employed to this end:

1. **Environment revisioning**: `EnvRev` is computed deterministically from the run identifier, configuration hash, data snapshot identifier, and master seed.
2. **Policy revisioning**: Any calibration state change that could alter a policy decision must result in an increment of `PolicyRev`, thereby preventing reuse of an obsolete decision cache.
3. **Batch-boundary updates**: Updates to audit, calibrator, and DIDS assignment are applied not in wall-clock completion order, but only to a sorted event sequence at batch boundaries.

Thanks to this architecture, the repository can simultaneously support concurrency, caching, and policy customization while maintaining replayability.

### 1.3 AdaptiveExecutor and the Single API

`DefaultBundle` uses `AdaptiveExecutor`, which determines the execution mode through a single `workers` parameter:

- `workers = 1`: `InlineExecutor` (sequential, zero overhead)
- `workers > 1`: `WorkerPoolExecutor` (fixed-size thread pool, batch-barrier synchronization)

`AdaptiveExecutor` determines thread availability at build time via the `#[cfg(imads_has_threads)]` gate. On native targets and on `wasm32-wasip1-threads` and `wasm32-wasip3`, the pool variant is enabled; on targets without thread support, such as `wasm32-unknown-unknown`, only the inline variant is used. This allows users to switch between single-threaded and multi-threaded execution through a single API without changing the generic bundle type.

### 1.4 Two-Axis Fidelity Ladder

On top of classical MADS, this implementation adds a fidelity ladder indexed by

\[
\phi = (\tau, S)
\]

where \(\tau\) denotes a solver tolerance scale and \(S\) denotes a Monte Carlo sample size. The default ladder policy (`StaircaseLadder`) generally produces a staircase of the form

\[
(\tau_1,S_1) \rightarrow (\tau_1,S_2) \rightarrow (\tau_2,S_2) \rightarrow \cdots
\]

That is, \(\tau\) moves from loose to tight and \(S\) moves from small to large. This design offers two advantages:

- Low-cost information is consumed first, and expensive truth evaluations are purchased late.
- **Same-\(S\), tighter-\(\tau\)** comparisons are made explicit, which directly benefits the paired-audit K learning that is central to this patch.

The engine treats the last level of the ladder as **TRUTH**. All preceding levels produce only `Estimates` and cannot directly update acceptance.

### 1.5 Search, Quantization, Poll

The default search policy is intentionally conservative. It combines deterministic local exploration around the incumbent with deterministic global exploration whose radius increases slowly as iterations proceed. All generated raw proposals are quantized onto the current mesh, and this process ensures canonical coordinate equality and cache coherence.

Mesh state is represented by `MeshGeometry`, whose key parameters are:

- `base_step`: the base lattice spacing \(\Delta_0\),
- `mesh_mul`: the current mesh multiplier,
- `refine_div`: an integer-based refinement divisor,
- `poll_step_mult`: a base-lattice-unit poll-step multiplier.

Accordingly, the current mesh spacing is

\[
\Delta = \Delta_0 \cdot \texttt{mesh\_mul}
\]

and refinement is performed via integer division, which structurally preserves the nested mesh property. The default poll operator (`DefaultPoll`) generates signed coordinate directions along each axis around the incumbent mesh point. The acceptance rule is sealed, but the poll itself remains transparent and simple in form.

## 2. Partial Information, Conservative Screening, and Acceptance

### 2.1 Partial Estimates

At fidelity levels preceding truth, the evaluator returns an `Estimates` object containing:

- An objective function estimate \(\hat f\) and its standard error,
- Constraint estimates \(\hat c_j\) and their standard errors,
- The fidelity bucket \(\phi=(\tau,S)\) that produced the estimates,
- A `tau_scale` used in the solver-induced bias model.

The default toy evaluator illustrates the intended contract well. Monte Carlo samples are deterministic with respect to \((x,\phi,\mathrm{env},k)\), and solver bias is expressed as a \(\tau\)-dependent deterministic perturbation. The engine assumes no further internal structure beyond this.

### 2.2 Margin Policy

The margin policy converts calibrator state into actual screening thresholds. In the current implementation, an early infeasible declaration for constraint \(j\) is made only when the following inequality is satisfied:

\[
\hat c_j - z_{\alpha_c}\,\hat\sigma_{c_j} - K_{c,j}(\phi)\,\tau_{\mathrm{scale}} > \delta^{\mathrm{rel}}_j.
\]

Three points are important here:

1. The test is **one-sided** and **conservative**.
2. The effective bias coefficient is no longer entirely global. Since this patch, the implementation first uses the **\((\tau,S)\)-bucket-specific** value `k_c_for(phi, j)` when available, falling back to the global coefficient only when it is not.
3. The threshold vector \(\delta^{\mathrm{rel}}\) is maintained per constraint and calibrated adaptively.

Objective function pruning is likewise structured as a calibration-aware conservative bound test:

\[
\mathrm{LCB}_f = \hat f - z_{\alpha_f}\hat\sigma_f - K_f(\phi)\,\tau_{\mathrm{scale}},
\qquad
\mathrm{UCB}_f = \hat f + z_{\beta_f}\hat\sigma_f + K_f(\phi)\,\tau_{\mathrm{scale}}.
\]

The current implementation uses `k_f_for(phi)` when available. Thus, objective pruning is not an ad hoc heuristic independent of calibration, but rather a conservative lower-bound test that directly reflects the calibration state.

### 2.3 Truth-Only Acceptance

The acceptance logic is intentionally sealed and adopts a **filter plus progressive barrier** structure. Non-truth results may influence scheduling and promotion, but they cannot insert points into the filter or finalize feasibility. Only `Truth` results can update acceptance.

This design strictly separates two actions:

- **Screening**: the act of early termination for candidates that are clearly dominated or clearly infeasible,
- **Acceptance**: the act of updating the feasible incumbent and filter only at terminal fidelity.

From a software coherence standpoint, this separation is critically important. Once partial estimates begin to directly modify acceptance structures, maintaining cache semantics, policy revisioning, and replayability simultaneously becomes drastically more difficult.

## 3. Audited Early Infeasible Cut and Paired-Audit K Second-Order Refinement

### 3.1 Why Audit Is Necessary

Early infeasible cuts are attractive from a cost standpoint but statistically risky. If cuts are overly aggressive, points that would prove feasible at truth may be prematurely eliminated. IMADS prevents this by employing a deterministic audit policy. Candidates selected for audit are not terminated immediately; instead, they are forcibly promoted to a higher fidelity.

The selector itself is hash-based and therefore reproducible. However, what is more fundamental is not the selector, but the **structure of the audit path**.

### 3.2 From Single Paired Samples to Multiple \(\tau\) Checkpoints

Prior to this patch, K learning was effectively a first-order version. A single early-cut estimate was associated with at most one paired checkpoint. The post-patch implementation substantially generalizes this.

When an audited cut occurs at ladder index `phi_idx_at_cut`, the engine collects all subsequent ladder indices satisfying the following conditions:

- The same candidate \(x\),
- The same Monte Carlo budget \(S\),
- A tighter solver tolerance \(\tau' < \tau\).

These indices are stored in `CandidateAuditOrigin::paired_phi_indices` and define a structured audit subpath. Consequently, a candidate does not simply jump to truth; instead, it can pass through multiple same-\(S\), tighter-\(\tau\) checkpoints, accumulating information along the way.

### 3.3 Structured K Accumulation

Upon reaching truth, the calibrator reconstructs a checkpoint sequence of the form

\[
(\tau^{(0)},S,\hat y^{(0)}) \, ,\, (\tau^{(1)},S,\hat y^{(1)}) \, ,\, \dots \, ,\, (\tau^{(r)},S,\hat y^{(r)})
\]

where \(\hat y\) collectively denotes objective and constraint estimates. The calibrator then forms interval pairs between adjacent checkpoints, rather than using only a single loose-versus-truth comparison. For each adjacent pair, the following tau-normalized positive deltas are computed:

\[
\frac{(\hat f^{\text{loose}}-\hat f^{\text{tight}})_+}{\tau^{\text{loose}}-\tau^{\text{tight}}},
\qquad
\frac{(\hat c_j^{\text{loose}}-\hat c_j^{\text{tight}})_+}{\tau^{\text{loose}}-\tau^{\text{tight}}}.
\]

These values are accumulated into three levels of statistical views:

1. **Global K samples**: used to update the engine-wide `k_f` and `k_c[j]`.
2. **Phi-bucketed K samples**: aggregated by the loose endpoint \((\tau,S)\) to update `k_by_phi`.
3. **Interval-bucketed K samples**: aggregated by \((\tau_{\text{loose}},\tau_{\text{tight}},S)\) and exposed as `k_interval_stats`.

When no structured same-\(S\) checkpoints are available, a cut-to-truth fallback is used. That is, functionality is preserved, but the preferred path is clearly distinguished as same-\(S\)-based interval accumulation.

### 3.4 Quantile-Smoothed K Update

The K update is conservative in two respects. First, only the positive part of the loose-minus-tight delta is accumulated, so the learned K is closer to an upper-envelope surrogate than to an unbiased difference estimator. Second, K is updated not as a mean but as an exponential moving average of a high empirical quantile:

\[
K^{\text{new}} = (1-\eta_K)K^{\text{old}} + \eta_K Q_q.
\]

Here, \(Q_q\) is the empirical \(q\)-quantile of recent samples. Given that the margin policy uses K inside a one-sided screening inequality, this choice is best interpreted as a design that tracks a **bias scale sufficiently large from a safety perspective**, rather than tracking the "typical" bias.

## 4. Refined DIDS Assignment Update

### 4.1 Meaning of \(a_j\)

The DIDS policy maintains an assignment vector of the form \(a_j \in \{1,\dots,L\}\), where \(L\) is the ladder length. Its meaning is straightforward:

> Early infeasible termination for constraint \(j\) is permitted only at fidelity levels whose 1-based ladder index is at least \(a_j\).

Therefore, larger \(a_j\) implies greater conservatism. The assignment policy ultimately mediates the cost--safety trade-off of "how high a fidelity must be purchased before a cut can be trusted?"

### 4.2 Recency-Weighted Base Rule

The first stage of the update estimates, for each constraint, an appropriate cutting level from recorded early infeasible outcomes. Since this patch, instead of simple frequency tallying, an exponential recency factor is applied and the base level is taken as a weighted upper quantile. This allows recent observations to naturally carry greater influence without a hard-truncation sliding window over older data.

### 4.3 Confidence-Aware False Infeasible Control

The second stage of the update uses the audit statistics maintained by the calibrator. For each ladder level and constraint, Wilson confidence bounds are computed for the following two quantities:

- The false infeasible rate,
- A precision signal indicating whether the originally flagged constraint was indeed violated at truth.

If the Wilson upper bound of the false infeasible rate is too large, or if the Wilson lower bound of precision is too low, \(a_j\) is raised. In other words, the code no longer reacts based solely on point estimates; instead, it reacts based on **conservative statistics that explicitly account for finite-sample uncertainty** -- this is the key contribution of the current patch.

### 4.4 Relaxation Logic

Conversely, the rule for lowering \(a_j\) has become considerably more stringent. Sufficient audit evidence must have accumulated at the level immediately below, the false infeasible upper bound must be well below the target, and the precision lower bound must be adequate at both the local and overall levels. In short, the new policy does not reduce conservatism merely because "the recent numbers look plausible," but permits relaxation **only when statistically justified**.

### 4.5 Interaction with \(\delta^{\mathrm{rel}}\)

The final fallback guard interprets the case where `delta_rel[j]` has grown notably large relative to baseline as a signal that "this constraint has become difficult to screen safely." When the ratio `delta_rel[j] / delta_baseline` grows sufficiently large, additional upward pressure is applied to \(a_j\). That is, the DIDS logic is not decoupled from the calibrator's false-infeasible control loop, but is reconnected to it.

## 5. Engine Semantics and Source Structure

### 5.1 Candidate Lifecycle

Each candidate is maintained as a `CandidateStageState`, which contains the following information:

- The current ladder index,
- The most recently obtained estimate,
- Audit flag and audit origin,
- A policy revision snapshot at the time of submission,
- An incumbent snapshot identifier,
- Terminal or resumable status.

This ensures that even when the engine processes a mixture of newly proposed candidates and candidates suspended from a previous iteration, it does not lose track of the premises under which subsequent decisions are valid.

### 5.2 Cache Separation

The repository separates the cache into two layers:

- `EvalCache`: a cache of expensive black-box estimates keyed by \((x,\phi,\mathrm{env\_rev})\),
- `DecisionCache`: a cache of policy-dependent materialized outcomes keyed by \((x,\phi,\mathrm{env\_rev},\mathrm{policy\_rev},\mathrm{tag})\).

This separation is not merely an implementation convenience. It is essential for preserving evaluation reuse even when policies change, while ensuring that policy-sensitive decisions are not replayed under an obsolete calibration regime.

### 5.3 Workspace Structure and Key Source Files

The codebase is organized as a Cargo workspace:

- **`imads-core/`**: Core engine crate
- **`imads-ffi/`**: C/C++ FFI (opaque handle + `cbindgen` header)
- **`imads-jvm/`**: JVM bindings via FFM (JDK 22+ Foreign Function & Memory API)
- **`imads-py/`**: Python bindings (PyO3/maturin, CPython + GraalPython)
- **`imads-wasm/`**: WASM bindings via WASI Component Model (wit-bindgen)

The core engine crate (`imads-core/src/`) is most easily understood when read in approximately the following order:

1. `types.rs`: Canonical state objects, fidelity types, cache keys, candidate lifecycle metadata.
2. `core/engine.rs`: Deterministic step-wise engine, `AdaptiveExecutor`, materialization logic, audit-path routing, policy-boundary updates.
3. `policies/margin.rs`: Conservative early infeasible test and objective bound computation.
4. `policies/calibrator.rs`: Delta control, paired-audit bookkeeping, K quantile update, bucketed statistics.
5. `policies/dids.rs`: Adaptive assignment rule for determining early infeasible termination levels.
6. `core/acceptance.rs`: Sealed filter-plus-progressive-barrier acceptance.
7. `policies/ladder.rs`, `policies/search.rs`, `core/poll.rs`: Ladder construction, candidate proposal, poll geometry.

### 5.4 Character of the Current Implementation

The current codebase is best understood as a **deterministic IMADS engine with a well-defined policy surface**. It already possesses substantial algorithmic and infrastructure content -- including calibrated screening, paired-audit K logic, resumable state transitions, and multi-language FFI support -- but it would be premature to present it as a complete replacement for large-scale mature DFO packages. The core value of this implementation lies in making the interactions among MADS-family search, multi-fidelity screening, determinism, and cache coherence explicit and auditable.
