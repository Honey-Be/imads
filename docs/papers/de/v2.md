# IMADS: Improved / Integrated Mesh Adaptive Direct Search

> **Anmerkung zur Abkurzung.** In diesem Repository wird **IMADS** als Oberbegriff verwendet, der sowohl **Improved Mesh Adaptive Direct Search** als auch **Integrated Mesh Adaptive Direct Search** umfasst.

## Zusammenfassung

Dieses Repository implementiert eine deterministische, ableitungsfreie Optimierungs-Engine (Derivative-Free Optimization Engine), die auf den Grundgedanken der Mesh Adaptive Direct Search (MADS)-Familie aufbaut, jedoch um moderne softwaretechnische Anforderungen herum reorganisiert wurde -- namentlich **Multi-Fidelity-basiertes fruehes Screening**, **Cache-Konsistenz**, **resumable Evaluation** sowie **Modularitaet der Policy-Schicht**. Die aktuelle Revision ist daher weniger als schlichte Reimplementierung eines monolithischen MADS-Solvers im klassischen Sinne zu verstehen, sondern vielmehr als eine **IMADS-Implementierung**, die eine \((\tau, S)\)-zweiachsige Fidelity-Leiter, stufenweise Kandidaten-Promotion, deterministische Audit-Selektion, Filter-plus-Progressive-Barrier-Akzeptanz sowie Online-Kalibrierung konservativer Screening-Schwellenwerte integriert.

Diese Implementierung ist um zwei Ziele herum organisiert. Erstens müssen alle Operationen, die eine Policy-Grenze (Policy Boundary) durchqueren, reproduzierbar sein. Zu diesem Zweck ist der Evaluation-Cache-Schluessel als \((x, \phi, \mathrm{env\_rev})\) definiert, waehrend der Decision-Cache-Schluessel diesen um \(\mathrm{policy\_rev}\) erweitert; saemtliche Policy-Updates werden ausschliesslich an deterministischen Batch-Grenzen durchgefuehrt. Zweitens werden kostenguenstige partielle Informationen aktiv zur Beschleunigung der Suche herangezogen, jedoch nicht zur endgueltigen Feststellung von Zulässigkeit oder Akzeptanz verwendet. Insbesondere staerkt der aktuelle Patch die Codebasis in zwei Richtungen substantiell: (i) **Paired-Audit K, Verfeinerung zweiter Ordnung**: Für denselben Punkt \(x\), dasselbe \(S\), werden mehrere straffere \(\tau\)-Checkpoints strukturiert akkumuliert, um K zu lernen. (ii) **Verfeinerte a-Aktualisierung**: Recency-Gewichtung, Wilson-Konfidenzintervalle, False-Infeasible-Statistiken und Praezisionssignale werden gemeinsam verwendet, um die Konservativitaet des fruehen Infeasible-Schnitts zu regulieren.

Ferner ist die aktuelle Codebasis als Cargo-Workspace strukturiert, wobei die Kern-Engine (`imads-core`) im Zentrum steht, ergaenzt durch Binding-Crates für C/C++ FFI (`imads-ffi`), JVM FFM (`imads-jvm`), Python/PyO3 (`imads-py`) und WASM Component Model (`imads-wasm`). Dadurch kann dieselbe Engine neben Rust auch in Python (CPython/GraalPython), Java, Kotlin, Scala 3, Clojure, TypeScript, C/C++ und weiteren Sprachumgebungen eingesetzt werden.

IMADS ist daher weniger als einfache Direktsuch-Implementierung zu verstehen, sondern vielmehr als eine deterministische, Policy-separierbare, Multi-Fidelity-MADS-Engine, die das Zusammenspiel von Reproduzierbarkeit, adaptivem Screening und konservativer ableitungsfreier Suche explizit offenlegt.

## 1. Algorithmische Übersicht

### 1.1 Problemformulierung

Der Gegenstand der Engine ist das folgende Black-Box-Optimierungsproblem mit Nebenbedingungen:

\[
\min_x f(x) \quad \text{subject to} \quad c_j(x) \le 0, \qquad j=1,\dots,m.
\]

Die grundlegende Annahme lautet, dass Ableitungen entweder nicht existieren oder, sofern sie existieren, nicht verwendet werden. Die Implementierung verfolgt keinen naiven, auf dem kontinuierlichen Raum operierenden Solver-Ansatz, sondern eine **Mesh-State**-zentrierte Struktur. Kandidatenpunkte werden als kanonische Koordinaten `XMesh` auf dem Basisgitter dargestellt, und kontinuierliche Koordinaten `XReal` werden ausschliesslich über eine geprufte Abbildung (Mapping) gewonnen. Dies steht im Einklang mit der grundlegenden MADS-Perspektive, konvergenz- und determinismuskritische Operationen nicht über Gleitkomma-Positionen, sondern über die Mesh-Geometrie zu definieren.

### 1.2 Deterministisches Ausführungsmodell

Der Kern dieser Codebasis behandelt Determinismus nicht als eine Eigenschaft, die "nach Moeglichkeit beibehalten" wird, sondern als **semantische Anforderung**. Dazu werden drei Mechanismen eingesetzt:

1. **Umgebungs-Revisionierung**: `EnvRev` wird deterministisch aus Run-Identifier, Konfigurations-Hash, Daten-Snapshot-Identifier und Master-Seed berechnet.
2. **Policy-Revisionierung**: Jede Änderung des Kalibrierungszustands, die Policy-Entscheidungen beeinflussen kann, fuehrt zwingend zu einer Erhöhung von `PolicyRev`, wodurch die Wiederverwendung veralteter Decision-Cache-Eintraege unterbunden wird.
3. **Batch-Grenz-Updates**: Aktualisierungen von Audit, Calibrator und DIDS-Assignment haengen nicht von der Wall-Clock-Completion-Reihenfolge ab, sondern werden ausschliesslich an Batch-Grenzen auf sortierte Ereignisfolgen angewendet.

Dank dieser Struktur kann das Repository Concurrency, Caching und Policy-Anpassung gleichzeitig unterstützen und dabei die Reproduzierbarkeit wahren.

### 1.3 AdaptiveExecutor und einheitliche API

`DefaultBundle` verwendet den `AdaptiveExecutor`, wobei der Ausführungsmodus durch den einzigen Parameter `workers` bestimmt wird:

- `workers = 1`: `InlineExecutor` (sequentiell, ohne Overhead)
- `workers > 1`: `WorkerPoolExecutor` (Thread-Pool fester Groesse, Batch-Barrier-Synchronisation)

Der `AdaptiveExecutor` bestimmt die Thread-Verfügbarkeit zur Build-Zeit über das Gate `#[cfg(imads_has_threads)]`. Auf nativen Targets sowie `wasm32-wasip1-threads` und `wasm32-wasip3` wird die Pool-Variante aktiviert; auf Targets ohne Thread-Unterstützung wie `wasm32-unknown-unknown` wird ausschliesslich die Inline-Variante verwendet. Dadurch können Nutzer zwischen Single-Thread- und Multi-Thread-Ausführung wechseln, ohne den generischen Bundle-Typ zu ändern -- über eine einheitliche API.

### 1.4 Zweiachsige Fidelity-Leiter

Über das klassische MADS hinaus fuehrt diese Implementierung eine Fidelity-Leiter ein, die durch

\[
\phi = (\tau, S)
\]

indiziert wird, wobei \(\tau\) die Solver-Toleranzskala und \(S\) die Monte-Carlo-Stichprobengroesse bezeichnet. Die Standard-Leiter-Policy (`StaircaseLadder`) erzeugt im Wesentlichen eine Treppenkonstruktion der Form

\[
(\tau_1,S_1) \rightarrow (\tau_1,S_2) \rightarrow (\tau_2,S_2) \rightarrow \cdots
\]

Das heisst, \(\tau\) bewegt sich von lose nach streng und \(S\) von klein nach gross. Dieses Design bietet zwei Vorteile:

- Kostenguenstige Informationen werden zuerst genutzt, waehrend teure Truth-Evaluierungen erst spaet erworben werden.
- Der Vergleich **gleichen \(S\), strengeren \(\tau\)** wird explizit hervorgehoben, was dem in diesem Patch zentralen Paired-Audit-K-Lernen direkt zugutekommt.

Die Engine betrachtet die letzte Stufe der Leiter als **TRUTH**. Alle vorherigen Stufen erzeugen lediglich `Estimates` und können die Akzeptanz nicht direkt aktualisieren.

### 1.5 Search, Quantisierung, Poll

Die Standard-Search-Policy ist bewusst konservativ gehalten. Sie kombiniert deterministische lokale Exploration in der Umgebung des Incumbent mit deterministischer globaler Exploration, deren Radius mit fortschreitender Iteration langsam waechst. Alle erzeugten Rohvorschlaege werden auf das aktuelle Mesh quantisiert; durch diesen Vorgang werden kanonische Koordinatengleichheit und Cache-Kohaerenz sichergestellt.

Der Mesh-Zustand wird durch `MeshGeometry` dargestellt, wobei die Kernparameter wie folgt lauten:

- `base_step`: Basis-Gitterabstand \(\Delta_0\),
- `mesh_mul`: aktueller Mesh-Multiplikator,
- `refine_div`: ganzzahlbasierter Refinement-Divisor,
- `poll_step_mult`: Poll-Step-Multiplikator in Einheiten des Basisgitters.

Somit ergibt sich der aktuelle Mesh-Abstand als

\[
\Delta = \Delta_0 \cdot \texttt{mesh\_mul}
\]

und das Refinement wird mittels ganzzahliger Division durchgefuehrt. Dadurch wird die Eigenschaft geschachtelter Meshes strukturell aufrechterhalten. Der Standard-Poll-Operator (`DefaultPoll`) erzeugt achsenparallele, vorzeichenbehaftete Koordinatenrichtungen um den Incumbent-Mesh-Punkt. Die Akzeptanzregel ist versiegelt (sealed), doch der Poll selbst bleibt transparent und in einfacher Gestalt.

## 2. Partielle Information, konservatives Screening und Akzeptanz

### 2.1 Partielle Schätzungen

Auf Fidelity-Stufen unterhalb der Truth liefert der Evaluator ein `Estimates`-Objekt zurueck. Dieses enthaelt:

- Zielfunktionsschaetzung \(\hat f\) und Standardfehler,
- Nebenbedingungsschaetzungen \(\hat c_j\) und Standardfehler,
- den Fidelity-Bucket \(\phi=(\tau,S)\), aus dem die Schätzung stammt,
- `tau_scale`, das im Solver-induzierten Bias-Modell verwendet wird.

Der Standard-Toy-Evaluator veranschaulicht den intendierten Vertrag praegnant: Monte-Carlo-Stichproben sind deterministisch bezueglich \((x,\phi,\mathrm{env},k)\), und der Solver-Bias wird als \(\tau\)-abhängige, deterministische Stoerung dargestellt. Die Engine setzt darueber hinaus keine weitere interne Struktur voraus.

### 2.2 Margin-Policy

Die Margin-Policy transformiert den Calibrator-Zustand in tatsaechliche Screening-Schwellenwerte. In der aktuellen Implementierung wird eine fruehe Infeasible-Erklaerung für Nebenbedingung \(j\) nur dann ausgesprochen, wenn die folgende Ungleichung erfuellt ist:

\[
\hat c_j - z_{\alpha_c}\,\hat\sigma_{c_j} - K_{c,j}(\phi)\,\tau_{\mathrm{scale}} > \delta^{\mathrm{rel}}_j.
\]

Dabei sind drei Aspekte wesentlich:

1. Der Test ist **einseitig (one-sided)** und **konservativ**.
2. Der effektive Bias-Koeffizient ist nicht mehr vollstaendig global. Nach diesem Patch verwendet die Implementierung, sofern verfügbar, zunächst den **\((\tau,S)\)-Bucket-spezifischen** Wert `k_c_for(phi, j)` und faellt nur bei dessen Abwesenheit auf den globalen Koeffizienten zurueck.
3. Der Schwellenwertvektor \(\delta^{\mathrm{rel}}\) wird je Nebenbedingung gefuehrt und adaptiv kalibriert.

Das Pruning der Zielfunktion ist ebenfalls als kalibrierungsbewusster, konservativer Schrankentest konstruiert:

\[
\mathrm{LCB}_f = \hat f - z_{\alpha_f}\hat\sigma_f - K_f(\phi)\,\tau_{\mathrm{scale}},
\qquad
\mathrm{UCB}_f = \hat f + z_{\beta_f}\hat\sigma_f + K_f(\phi)\,\tau_{\mathrm{scale}}.
\]

Die aktuelle Implementierung verwendet, sofern moeglich, `k_f_for(phi)`. Somit ist das Objective-Pruning kein von der Kalibrierung unabhängiger Ad-hoc-Heuristik, sondern ein konservativer Untergrenzentest, der den Kalibrierungszustand direkt widerspiegelt.

### 2.3 Truth-Only-Akzeptanz

Die Akzeptanzlogik ist bewusst versiegelt und folgt einer **Filter-plus-Progressive-Barrier**-Struktur. Non-Truth-Ergebnisse können Scheduling und Promotion beeinflussen, jedoch keinen Punkt in den Filter einfuegen oder Zulässigkeit endgueltig feststellen. Ausschliesslich `Truth`-Ergebnisse können die Akzeptanz aktualisieren.

Dieses Design trennt die folgenden beiden Handlungen strikt:

- **Screening**: das fruehe Abbrechen von Kandidaten, die offensichtlich dominiert oder offensichtlich infeasible sind,
- **Akzeptanz**: die ausschliesslich bei terminaler Fidelity erfolgende Aktualisierung des Feasible-Incumbent und des Filters.

Aus Sicht der Software-Konsistenz ist diese Trennung von grosser Bedeutung. Sobald partielle Schätzungen beginnen, die Akzeptanzstruktur direkt zu veraendern, wird es aeusserst schwierig, Cache-Semantik, Policy-Revisionierung und Reproduzierbarkeit gleichzeitig aufrechtzuerhalten.

## 3. Audited Early Infeasible Cut und Paired-Audit K, Verfeinerung zweiter Ordnung

### 3.1 Warum Audits notwendig sind

Der fruehe Infeasible-Schnitt ist aus Kostensicht attraktiv, jedoch statistisch riskant. Ist der Schnitt zu aggressiv, können Punkte, die sich bei Truth als zulässig erweisen wuerden, vorzeitig eliminiert werden. IMADS begegnet diesem Risiko durch eine deterministische Audit-Policy. Für das Audit ausgewaehlte fruehe Schnitte werden nicht sofort beendet, sondern zwangsweise auf eine hoehere Fidelity-Stufe befoebert.

Der Selektor selbst ist Hash-basiert und daher reproduzierbar. Wesentlicher als der Selektor ist jedoch die **Struktur des Audit-Pfades**.

### 3.2 Von einzelnen gepaarten Stichproben zu multiplen \(\tau\)-Checkpoints

Vor diesem Patch entsprach das K-Lernen im Wesentlichen einer Version erster Ordnung. Eine einzelne Early-Cut-Schätzung war mit hoechstens einem gepaarten Checkpoint verknuepft. Nach dem Patch verallgemeinert die Implementierung dies substantiell.

Wenn ein auditierter Schnitt am Leiter-Index `phi_idx_at_cut` auftritt, sammelt die Engine alle nachfolgenden Indizes auf der Leiter, die folgende Bedingungen erfuellen:

- derselbe Kandidat \(x\),
- dasselbe Monte-Carlo-Budget \(S\),
- eine strengere Solver-Toleranz \(\tau' < \tau\).

Diese Indizes werden in `CandidateAuditOrigin::paired_phi_indices` gespeichert und definieren einen strukturierten Audit-Unterpfad. Folglich springt der Kandidat nicht einfach zur Truth, sondern kann mehrere Same-\(S\)-, Tighter-\(\tau\)-Checkpoints durchlaufen und dabei Information akkumulieren.

### 3.3 Strukturierte K-Akkumulation

Bei Erreichen der Truth rekonstruiert der Calibrator eine Checkpoint-Folge der Gestalt

\[
(\tau^{(0)},S,\hat y^{(0)}) \, ,\, (\tau^{(1)},S,\hat y^{(1)}) \, ,\, \dots \, ,\, (\tau^{(r)},S,\hat y^{(r)})
\]

wobei \(\hat y\) sowohl Zielfunktions- als auch Nebenbedingungsschaetzungen bezeichnet. Anschliessend verwendet der Calibrator nicht lediglich einen einzelnen Loose-vs-Truth-Vergleich, sondern bildet Intervallpaare zwischen benachbarten Checkpoints. Für jedes benachbarte Paar wird das folgende Tau-normalisierte positive Delta berechnet:

\[
\frac{(\hat f^{\text{loose}}-\hat f^{\text{tight}})_+}{\tau^{\text{loose}}-\tau^{\text{tight}}},
\qquad
\frac{(\hat c_j^{\text{loose}}-\hat c_j^{\text{tight}})_+}{\tau^{\text{loose}}-\tau^{\text{tight}}}.
\]

Diese Werte werden in drei statistische Sichten akkumuliert:

1. **Globale K-Stichproben**: Verwendet zur Aktualisierung von `k_f`, `k_c[j]` auf Engine-Gesamtebene.
2. **Phi-Bucket-bezogene K-Stichproben**: Gesammelt nach dem Loose-Endpunkt \((\tau,S)\) und zur Aktualisierung von `k_by_phi` verwendet.
3. **Intervall-Bucket-bezogene K-Stichproben**: Gesammelt nach \((\tau_{\text{loose}},\tau_{\text{tight}},S)\) und über `k_interval_stats` exponiert.

Wenn keinerlei strukturierte Same-\(S\)-Checkpoints vorliegen, wird ein Cut-to-Truth-Fallback verwendet. Die Funktionalitaet bleibt also erhalten, wobei der bevorzugte Pfad eindeutig als Same-\(S\)-basierte Intervallakkumulation gekennzeichnet ist.

### 3.4 Quantil-geglaettete K-Aktualisierung

Die K-Aktualisierung ist in zweifacher Hinsicht konservativ. Erstens wird ausschliesslich der positive Anteil des Loose-minus-Tight-Deltas akkumuliert, weshalb das gelernte K eher einem Upper-Envelope-Surrogat als einem unverzerrten Differenzschaetzer entspricht. Zweitens wird K nicht als Mittelwert, sondern als exponentiell gleitender Durchschnitt eines hohen empirischen Quantils aktualisiert:

\[
K^{\text{new}} = (1-\eta_K)K^{\text{old}} + \eta_K Q_q.
\]

Dabei ist \(Q_q\) das empirische \(q\)-Quantil der juengsten Stichproben. Beruecksichtigt man, dass die Margin-Policy K innerhalb einer einseitigen Screening-Ungleichung verwendet, so ist diese Wahl als Entwurfsentscheidung zu verstehen, die nicht den "typischen" Bias verfolgt, sondern eine **sicherheitsseitig hinreichend grosse Bias-Skala** nachzuhalten sucht.

## 4. Verfeinerte DIDS-Assignment-Aktualisierung

### 4.1 Bedeutung von \(a_j\)

Die DIDS-Policy unterhalt einen Assignment-Vektor der Form \(a_j \in \{1,\dots,L\}\), wobei \(L\) die Leiterlaenge bezeichnet. Die Semantik ist unmittelbar:

> Die fruehe Infeasible-Terminierung für Nebenbedingung \(j\) ist nur auf Fidelity-Stufen zulässig, deren 1-basierter Leiter-Index mindestens \(a_j\) betraegt.

Je größer \(a_j\), desto konservativer die Policy. Die Assignment-Policy vermittelt letztlich das Kosten-Sicherheits-Problem: "Wie hohe Fidelity muss erworben werden, bevor einem Schnitt vertraut werden darf?"

### 4.2 Recency-gewichtete Basisregel

Die erste Stufe der Aktualisierung schätzt aus den aufgezeichneten Early-Infeasible-Ergebnissen das je Nebenbedingung angemessene Cutting-Level. Nach dem Patch wird nicht mehr eine einfache Haeufigkeitsauszaehlung durchgefuehrt, sondern es wird ein exponentieller Recency-Faktor zugewiesen und anschliessend das gewichtete obere Quantil als Basis-Level bestimmt. Dadurch erhalten juengere Beobachtungen auf natuerliche Weise größeren Einfluss, ohne dass ein hart abschneidendes Schiebefenster erforderlich waere.

### 4.3 Konfidenz-bewusste False-Infeasible-Kontrolle

Die zweite Stufe der Aktualisierung nutzt die vom Calibrator gefuehrten Audit-Statistiken. Für jede Leiterstufe und Nebenbedingung werden Wilson-Konfidenzschranken für die folgenden beiden Groessen berechnet:

- die False-Infeasible-Rate,
- das Praezisionssignal, d.h. ob die urspruenglich markierte Nebenbedingung auch bei Truth tatsaechlich verletzt war.

Ist die obere Wilson-Schranke der False-Infeasible-Rate zu gross oder die untere Wilson-Schranke der Praezision zu niedrig, wird \(a_j\) angehoben. Der entscheidende Beitrag dieses Patches besteht also darin, dass der Code nicht mehr lediglich auf Punktschaetzungen reagiert, sondern auf **konservative statistische Kenngroessen, die die endliche Stichprobengroesse explizit beruecksichtigen**.

### 4.4 Relaxationslogik

Umgekehrt ist auch die Regel zur Absenkung von \(a_j\) deutlich strenger geworden. Die unmittelbar darunterliegende Stufe muss hinreichend Audit-Evidenz aufweisen; die obere False-Infeasible-Schranke muss genuegend unterhalb des Zielwerts liegen; und die untere Praezisionsschranke muss sowohl auf lokaler als auch auf Gesamtebene angemessen sein. Kurzum: Die neue Policy senkt die Konservativitaet nicht bereits dann, wenn "die juengsten Zahlen plausibel erscheinen", sondern gestattet eine Relaxation **ausschliesslich bei statistischer Rechtfertigung**.

### 4.5 Wechselwirkung mit \(\delta^{\mathrm{rel}}\)

Ein letzter Fallback-Guard interpretiert den Fall, dass `delta_rel[j]` relativ zur Baseline ungewoehnlich gross geworden ist, als Signal dafuer, dass "diese Nebenbedingung schwer sicher zu screenen geworden ist". Übersteigt das Verhaeltnis `delta_rel[j] / delta_baseline` einen hinreichenden Wert, wird zusaetzlicher Aufwaertsdruck auf \(a_j\) ausgeuebt. Somit operiert die DIDS-Logik nicht isoliert vom False-Infeasible-Kontrollkreis des Calibrators, sondern ist wieder mit diesem verbunden.

## 5. Engine-Semantik und Quellcodestruktur

### 5.1 Kandidaten-Lebenszyklus

Jeder Kandidat wird in einem `CandidateStageState` gefuehrt, der folgende Informationen enthaelt:

- aktueller Leiter-Index,
- zuletzt gesicherte Schätzung,
- Audit-Flag und Audit-Herkunft,
- Snapshot der Policy-Revision zum Zeitpunkt der Einreichung,
- Incumbent-Snapshot-Identifier,
- Terminal- oder Resumable-Status.

Dadurch kann die Engine neu vorgeschlagene Kandidaten und aus frueheren Iterationen unterbrochene Kandidaten gemischt verarbeiten, ohne dass die Grundlagen, auf denen spätere Entscheidungen gueltig sind, verloren gehen.

### 5.2 Cache-Trennung

Das Repository unterteilt den Cache in zwei Schichten:

- `EvalCache`: Cache für teure Black-Box-Schätzungen mit dem Schluessel \((x,\phi,\mathrm{env\_rev})\),
- `DecisionCache`: Cache für Policy-abhängige, materialisierte Ergebnisse mit dem Schluessel \((x,\phi,\mathrm{env\_rev},\mathrm{policy\_rev},\mathrm{tag})\).

Diese Trennung ist keine blosse Implementierungsbequemlichkeit. Sie ist notwendig, um die Wiederverwendung von Evaluierungen auch bei Policy-Änderungen zu ermoeglichen und gleichzeitig zu verhindern, dass Policy-sensitive Entscheidungen unter einem veralteten Kalibrierungsregime reproduziert werden.

### 5.3 Workspace-Struktur und wesentliche Quelldateien

Die Codebasis ist als Cargo-Workspace organisiert:

- **`imads-core/`**: Kern-Engine-Crate
- **`imads-ffi/`**: C/C++ FFI (opaque Handle + `cbindgen`-Header)
- **`imads-jvm/`**: JVM-Binding (JDK 22+ FFM/Panama API)
- **`imads-py/`**: Python-Binding (PyO3/maturin, CPython + GraalPython)
- **`imads-wasm/`**: WASM-Binding (WASI Component Model, wit-bindgen)

Das Kern-Engine-Crate (`imads-core/src/`) erschliesst sich am besten in folgender Lesereihenfolge:

1. `types.rs`: Kanonische Zustandsobjekte, Fidelity-Typen, Cache-Schluessel, Kandidaten-Lebenszyklus-Metadaten.
2. `core/engine.rs`: Deterministische, schrittweise Engine, `AdaptiveExecutor`, Materialisierungslogik, Audit-Pfad-Routing, Policy-Boundary-Update.
3. `policies/margin.rs`: Konservativer fruehe-Infeasibility-Test und Zielfunktions-Schrankenberechnung.
4. `policies/calibrator.rs`: Delta-Kontrolle, Paired-Audit-Buchfuehrung, K-Quantil-Aktualisierung, Bucket-bezogene Statistiken.
5. `policies/dids.rs`: Adaptive Assignment-Regel zur Festlegung des fruehen Infeasible-Terminierungslevels.
6. `core/acceptance.rs`: Versiegelte Filter-plus-Progressive-Barrier-Akzeptanz.
7. `policies/ladder.rs`, `policies/search.rs`, `core/poll.rs`: Leiter-Konstruktion, Kandidatenvorschlag, Poll-Geometrie.

### 5.4 Charakter der aktuellen Implementierung

Die aktuelle Codebasis ist am treffendsten als **deterministische IMADS-Engine mit wohldefinierter Policy-Oberflaeche** zu verstehen. Sie verfügt bereits in erheblichem Umfang über substantielle algorithmische und infrastrukturelle Inhalte -- kalibriertes Screening, Paired-Audit-K-Logik, resumable Zustandsuebergaenge, Multi-Sprach-FFI-Unterstützung --, doch waere es überzogen, sie umstandslos als vollstaendigen Ersatz für ein ausgereiftes, grossskaliges DFO-Paket zu praesentieren. Der zentrale Wert dieser Implementierung liegt darin, das Zusammenspiel von MADS-artiger Suche, Multi-Fidelity-Screening, Determinismus und Cache-Konsistenz in einer expliziten und nachpruefbaren Form offenzulegen.
